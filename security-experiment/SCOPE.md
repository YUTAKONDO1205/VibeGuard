# 実験項目スコープ選定（SCOPE）

対象: VibeGuard **サービス** `tool 0.1.3 / engine 0.1.0`、47 rules / 8 languages、4チャネル
（CLI・GitHub Action・VS Code・Chrome）。共有エンジンは `packages/analyzer-core`、
ルールは正規表現ベース（[`packages/rules/src/matcher-utils.ts`](../packages/rules/src/matcher-utils.ts) の `runRegex`）。

---

## 1. 脅威モデル（Threat Model）

「で、攻撃者は誰で、何を得るのか」を先に固定する。本実験が想定する攻撃者・資産・信頼境界。

### 資産（守るもの）
- **利用者のソースコードの機密性** — PRIVACY.md は「コードは端末外に出ない」と主張。
- **スキャナの可用性** — CI ゲート / IDE 保存時スキャンが固まらない・落ちないこと。
- **検出判定の完全性（integrity）** — 本物の脆弱性が「無い」と誤って見えないこと。
- **利用者のホスト／ブラウザ** — 拡張機能・Action が過剰権限や任意実行を持たないこと。

### 攻撃者モデル
| # | 攻撃者 | 能力 | 狙い |
|---|---|---|---|
| T1 | **悪意あるスキャン対象コード**（被スキャンファイルを送り込める） | VibeGuard に食わせる入力を完全制御 | スキャナをハング/クラッシュさせる（ReDoS, A1/A5）、検出をすり抜ける（B1/B3） |
| T2 | **検出を逃れたい開発者/AIエージェント** | コードの書き方を制御 | 脆弱性を残したまま「VibeGuard緑」を得る（B1/B2/B3） |
| T3 | **ネットワーク盗聴/サプライチェーン** | 配布物・依存・CI を観測 | コード持ち出し（A2）、依存経由の侵害（A4） |
| T4 | **悪意あるWebページ**（Chrome拡張利用時） | 被害者が訪れるページのDOMを制御 | 拡張の権限を悪用しコード/権限を奪う（A3） |

### 信頼境界
- 被スキャンコード = **常に未信頼**。エンジンは未信頼入力に対し有界時間・有界メモリで
  終了し、判定を曲げられてはならない。
- 拡張のホスト権限 `<all_urls>` = 利用者が明示スキャンした時のみページ内容を読む（PRIVACY.md主張）。

---

## 2. 採否方針

- **In scope**: 上記 T1–T4 に直接対応し、VibeGuard 自リポジトリ内で再現可能な実験。
- **Out of scope（やらない）**:
  - 第三者の本番システム・配信ストア基盤・GitHub/Google のインフラへの攻撃。
  - 公開ストアに出ている拡張の実利用者を対象にした調査。
  - 0day の武器化・実害を与えるエクスプロイト公開。検証は PoC（ハング再現・検出漏れ提示）に留める。
  - 既存 E1–E6 が既に扱う機能品質（再掲はするが再測定しない）。

---

## 3. 実験項目（2トラック8項目）

各項目: **仮説 / 手法 / 指標 / 成果物 / 優先度**。優先度は P0（サービスの安全性主張を支える必須）
> P1（あると強い）> P2（余力があれば）。

### Track A — サービス自身の安全性

#### A1. ReDoS / 計算量耐性 〔P0〕
- **対応脅威**: T1
- **仮説**: 47本の正規表現の一部は、未信頼入力に対し入力長に非線形（指数/二次）な
  実行時間を示し、`runRegex` には**時間・入力サイズの上限が無い**ため、単一ファイルで
  スキャン全体をハングさせられる。
- **手法**:
  1. 全ルールの正規表現を抽出し、静的にネスト量化子・交代の重複等
     （catastrophic backtracking パターン）を `recheck` / `vuln-regex-detector` 等で判定。
  2. 危険候補に対し攻撃文字列を生成し、入力長 n を増やしながら CLI 実測
     （n vs wall-time 曲線、線形 / 二次 / 指数を分類）。
  3. 4チャネル共有エンジンなので、CLI で再現＝全チャネルで再現を主張（E1の一貫性で裏取り）。
- **指標**: (a) 非線形ルール数 / 47、(b) 1ファイルでスキャンを T 秒超にできる最小入力長、
  (c) DESIGN.ja.md §11.1 の性能契約（3s/1s/5min）を破れるか yes/no。
- **成果物**: `track-a-tool-security/a1-redos/` に攻撃コーパス＋ n–time 表＋修正提案
  （per-match タイムアウト or 入力長キャップ or 安全な書き換え）。
- **サービス上の意味**: 「セキュリティサービス自身の可用性」を定量化。緩和策の前後比較で
  “見つけて直した”という改善サイクルに繋がる。

#### A2. ゼロ送信（no-egress）主張の実証 〔P0〕
- **対応脅威**: T3
- **仮説**: PRIVACY.md の「コードは端末外に出ない／ネットワークアクセス無し」は、
  静的にも動的にも検証可能で、4チャネルすべてで egress = 0。
- **手法**:
  1. **静的**: 配布バンドル（CLI dist / Chrome dist / VS Code vsix）に `fetch`/`XMLHttpRequest`/
     `node:http(s)`/`net`/`dns` 等のネットワークシンクが無いことを grep + 依存グラフで確認。
     Chrome は `manifest.json` に `host_permissions` 由来の送信経路が無いことも確認。
  2. **動的**: ネットワーク遮断 or プロキシ監視下（mitmproxy / `--inspect` + DevTools Network /
     Windows なら Fiddler）で、サンプルをスキャンし送信パケット 0 を観測。
- **指標**: 静的シンク数 0、動的キャプチャでの外向きリクエスト 0（チャネル×シナリオの表）。
- **成果物**: `a2-egress/` に静的監査結果＋キャプチャログ＋手順（第三者再現可能に）。
- **サービス上の意味**: 「ローカル完結」を“主張”でなく“実証済みの性質”に格上げ。
  プライバシーを掲げるサービスの信頼性に直結。

#### A3. Chrome 拡張の権限・攻撃面監査 〔P1〕
- **対応脅威**: T4
- **仮説**: Manifest V3 の権限（`activeTab`,`scripting`,`<all_urls>` 等）は最小権限で、
  悪意あるページから拡張経由でコード/権限を奪う経路が無い（content-script 注入・
  `storage` ハンドオフ・eval 不使用）。
- **手法**: manifest 静的監査（権限の必要性マッピング）、background/side-panel/content の
  メッセージ経路レビュー、`storage.session` ハンドオフのライフタイム確認、CSP と
  `eval`/`Function` 不使用の確認、`<all_urls>` が「明示スキャン時のみ読む」実装になっているか追跡。
- **指標**: 過剰権限件数、未検証の信頼境界クロス件数、危険API使用件数（いずれも 0 目標）。
- **成果物**: `a3-chrome-perms/` に権限表＋データフロー図＋指摘。

#### A4. サプライチェーン / 依存・Action 監査 〔P1〕
- **対応脅威**: T3
- **仮説**: 配布物のサプライチェーンリスクは管理されている（既知脆弱依存・過剰な
  Action 権限・未ピン依存が無い）。
- **手法**: `npm audit` / SBOM（CycloneDX）生成、`action.yml` と CI workflow の `permissions:`
  最小化確認、サードパーティ Action の SHA ピン状況、リリース成果物の出所（provenance）。
- **指標**: high+ 脆弱依存数、過剰権限ワークフロー数、未ピン Action 数。
- **成果物**: `a4-supplychain/` に SBOM＋監査表。
- **備考**: 既存 `.github/workflows/security-scan.yml` の self-scan と**役割が異なる**
  （あれはVibeGuardのルールで自スキャン。こちらは依存・権限・出所のサプライチェーン軸）。

#### A5. 未信頼入力ファジング（クラッシュ/ハング耐性） 〔P1〕
- **対応脅威**: T1
- **仮説**: 巨大・バイナリ・不正エンコーディング・極端なネスト等の異常ファイルでも、
  エンジンは無限ループ/未捕捉例外/メモリ枯渇を起こさず graceful に終了する。
- **手法**: CLI を入口に、生成/変異ファジング（巨大行、NUL、無効UTF-8、深いネスト、
  100MB級）。A1（ReDoS）と相補：A1は時間計算量、A5は堅牢性全般。
- **指標**: クラッシュ/ハング/未捕捉例外の件数、最大メモリ、回復可否。
- **成果物**: `a5-fuzz/` にコーパス＋失敗ケース＋修正提案。

### Track B — 敵対的検出耐性

#### B1. 意味保存難読化による回避 〔P0〕
- **対応脅威**: T1/T2
- **仮説**: 正規表現ベース検出は、**脆弱性を生かしたまま**の意味保存変換
  （文字列連結 `e"+"val`、`getattr`/エイリアス経由呼び出し、エンコード、空白/コメント挿入、
  動的属性アクセス）で容易に回避でき、検出率が大きく落ちる。
- **手法**: 既存 `samples/vulnerable` と `test_problem` の各脆弱性に、ルール別に
  k 種の意味保存変換を施した変異版を生成。元と変異版で検出有無を比較。
  「変換後もコードは依然として脆弱（実行可能性は最小PoCで担保）」を明示。
- **指標**: ルール別 evasion 率 = 回避できた変換 / 試行、全体の検出率低下（before→after）。
- **成果物**: `b1-evasion/` に変換カタログ＋検出マトリクス。
- **サービス上の意味**: regex検出の限界を定量化＝「だから context-confidence や将来の AST/semgrep
  深掘り（DESIGN の deep mode 予定）が要る」というロードマップ判断の根拠になる。
  限界を正直に把握することが、次の改善の出発点。

#### B2. AI生成の敵対コーパス 〔P1〕
- **対応脅威**: T2
- **仮説**: VibeGuard が標的とする「AI生成コード」自体を AI に作らせ、かつ
  「VibeGuardを通る脆弱コードを書け」と指示すると、検出をすり抜ける現実的な脆弱コードが得られる。
- **手法**: LLM に (a) 素の脆弱コード、(b) 「検出回避を意識した」脆弱コードを生成させ、
  両者の検出率を比較。生成物の脆弱性は人手 or 簡易実行で確認。倫理上、生成は自リポ検証目的に限定。
- **指標**: 素 vs 回避指示での検出率差、B1 の手動変換との一致度。
- **成果物**: `b2-ai-adversarial/` に生成プロンプト・コーパス・検出結果。

#### B3. context-confidence / suppression の悪用 〔P0・自作機能への攻撃〕
- **対応脅威**: T2
- **仮説**: VibeGuard の目玉である **context-window confidence 補正**
  （コメント/docstring/テストパスの一致を格下げ）と `vibeguard:disable-next-line` 抑制は、
  攻撃者が**本物の脆弱性を「テスト/フィクスチャ風パス」や「docstring偽装」に置いて
  action閾値未満へ格下げ**することで、隠蔽に悪用できる。
- **手法**:
  1. 同一の本物脆弱性を「通常パス」vs「`*/tests/*`・`*_mock.*`・`conftest.py` 等の偽装パス」に置き、
     confidence の格下げで medium 閾値を割るかを比較
     （[`packages/analyzer-core/src/confidence.ts`](../packages/analyzer-core/src/confidence.ts) の挙動を逆用）。
  2. docstring/ブロックコメント内に**実行される**コードを紛れ込ませる言語別トリック
     （例: 文字列終端の悪用）で格下げを誘発できるか。
  3. `vibeguard:disable-next-line` の濫用（AIが「とりあえず抑制」を書く）頻度の議論。
- **指標**: 格下げで action閾値を割れた脆弱性数 / 試行、偽装パス検出の有無。
- **成果物**: `b3-suppression-abuse/` に攻撃ケース＋（必要なら）緩和案
  （例: 高severity critical は test パスでも格下げしない、抑制コメントの監査ログ化）。
- **サービス上の意味**: **自分の売り機能を自分で攻める**ことで、context-confidence を
  「無批判な利点」でなく「トレードオフを把握した上での設計判断」に高められる。緩和策がそのまま
  プロダクト改善になる。

## 4. 優先度サマリ・推奨順

| 項目 | 優先度 | 工数感 | 推奨理由 |
|---|---|---|---|
| A1 ReDoS | **P0** | 中 | 実在の攻撃面（時間上限無し）。定量化＋緩和で“発見して直した”話になる |
| A2 ゼロ送信実証 | **P0** | 小〜中 | PRIVACY主張の実証。低工数で信頼性インパクト大 |
| B1 回避難読化 | **P0** | 中 | regex検出の限界を正直に定量化。改善ロードマップの核 |
| B3 context-confidence悪用 | **P0** | 中 | 自作目玉機能への敵対検証。最も差別化される |
| A3 Chrome権限 | P1 | 小 | 監査中心で軽い。MV3の正当性確認 |
| A4 サプライチェーン | P1 | 小 | ツール化（npm audit/SBOM）で軽い |
| A5 ファジング | P1 | 中 | A1の相補。堅牢性の網羅 |
| B2 AI敵対コーパス | P1 | 中 | B1を“AI生成”文脈で補強（本サービスの動機と直結） |

**推奨**: まず **P0 4本（A1・A2・B1・B3）** で骨子を立てる。これだけで
「自己安全性（A1可用性 + A2機密性） × 検出の限界と自機能の堅牢性（B1 + B3）」という
一貫したセキュリティ評価ストーリーが揃う。P1 は余力次第で充実。

## 5. 倫理・再現性

- 全実験は **VibeGuard 自リポジトリ（自サービス）**を対象。第三者システムへの攻撃・
  実利用者調査は含まない。
- 攻撃PoCは「ハング再現」「検出漏れ提示」までに留め、武器化コードは公開しない。
- E1–E6 と同様、各項目を**番号付きスクリプト＋期待値**で追跡可能にし、reference環境
  （EVALUATION.md と同一: Win11 / i5-12400 / Node v24）を明記。性能依存指標（A1/A5の時間）は
  ハードウェア依存を注記。
