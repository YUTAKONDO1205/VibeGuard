# 実験項目スコープ選定（SCOPE）

対象: VibeGuard **サービス** `tool 0.1.3 / engine 0.1.0`、47 rules / 8 languages、4チャネル
（CLI・GitHub Action・VS Code・Chrome）。共有エンジンは `packages/analyzer-core`、
ルールは正規表現ベース（[`packages/rules/src/matcher-utils.ts`](../packages/rules/src/matcher-utils.ts) の `runRegex`）。

本書は「自リポジトリの監査メモ」ではなく、**第三者が再現でき・一般化可能な知見を生む
研究設計**として書く。具体的には、研究設問(RQ)・新規性の主張・他SASTを含むベースライン比較・
形式的な指標定義・妥当性への脅威・再現可能な成果物を備える。

---

## 0. 研究としての位置づけ

### 中心命題（thesis）
「AI生成コードを標的とする**パターンベース（正規表現）SAST**」には、従来の SAST 評価
（再現率・FP率）では測られてこなかった2つの未検証リスクがある：

1. **検出器そのものが攻撃面になる** — ルール正規表現の計算量爆発（ReDoS）等で、
   未信頼な被スキャン入力によって CI/IDE のスキャンを可用性侵害できる。
2. **意味保存変換に対して脆い** — コードの意味（＝脆弱性）を保ったままの表層的書き換え
   （AIアシスタントが言い換え時に自然に生む類）で、検出が系統的に回避される。

VibeGuard を題材に、この2リスクを **定量化し・他SASTと比較し・再利用可能なベンチマークとして
公開できる形**にまとめる。VibeGuard 単体の良し悪しではなく、「パターンベースSASTの一般的性質」
として一般化することを狙う（だからベースライン比較を全実験に組み込む）。

### 研究設問（RQ）
- **RQ1（可用性・自己攻撃面）**: パターンベースSASTのルール集合は、未信頼入力により
  非線形時間でハングさせられるか。VibeGuard 固有か、他SAST（Semgrep等）にも見られるか。 → A1, A5
- **RQ2（機密性・主張の検証可能性）**: 「コードを端末外に出さない」というプライバシー主張は、
  静的＋動的に**反証可能な手続き**で検証できるか。検証方法論として一般化できるか。 → A2, A3, A4
- **RQ3（完全性・回避耐性）**: 意味保存変換は検出率をどれだけ下げるか。変換の**分類体系
  （taxonomy）**を作れるか。回避のしやすさはツール種別（正規表現 vs データフロー）でどう違うか。 → B1, B2
- **RQ4（自己防御機構のトレードオフ）**: VibeGuard 独自の context-window confidence 補正と
  抑制機構は、検出精度を上げる一方で**攻撃者による隠蔽**に悪用されうるか。その境界はどこか。 → B3

### 貢献（contributions）— 査読で「新しい知見」と言える単位
- **C1: 敵対的ベンチマーク（artifact）** — 意味保存変換のカタログ＋脆弱コーパスの変異版を、
  ツール非依存の再利用可能データセットとして公開（`b1-evasion/`）。
- **C2: 横断測定** — VibeGuard・Bandit・Semgrep（＋可能なら CodeQL）を**同一の敵対条件**で
  比較し、回避耐性がツール設計（パターン vs データフロー）にどう依存するかを示す。
- **C3: SASTルールの ReDoS という観点** — 「脆弱性を探す正規表現自体が ReDoS 源」という
  自己言及的リスクを、複数SASTのルール集合に対し体系測定（C2と同じ横断軸）。
- **C4: 自己防御機構の二面性** — context-confidence 補正の「FP削減」効果と「隠蔽悪用」リスクを
  同一指標で測り、緩和案（severity依存の格下げ抑止等）を定量評価。

> 単独ツールの監査に留めないために、**A1・A2・B1・B3 はベースラインSASTを必ず併走**させる。
> これが「VibeGuard が〜」を「パターンベースSAST一般が〜」に格上げする鍵。

## 1. 脅威モデル（Threat Model）

### 資産（守るもの）
- **利用者のソースコードの機密性** — PRIVACY.md は「コードは端末外に出ない」と主張。
- **スキャナの可用性** — CI ゲート / IDE 保存時スキャンが固まらない・落ちないこと。
- **検出判定の完全性（integrity）** — 本物の脆弱性が「無い」と誤って見えないこと。
- **利用者のホスト／ブラウザ** — 拡張機能・Action が過剰権限や任意実行を持たないこと。

### 攻撃者モデル
| # | 攻撃者 | 能力 | 狙い | RQ |
|---|---|---|---|---|
| T1 | 悪意ある被スキャンコード | 入力を完全制御 | スキャナをハング/クラッシュ | RQ1 |
| T2 | 検出を逃れたい開発者/AIエージェント | コードの書き方を制御 | 脆弱性を残し「緑」を得る | RQ3, RQ4 |
| T3 | 盗聴/サプライチェーン | 配布物・依存・CIを観測 | コード持ち出し・依存侵害 | RQ2 |
| T4 | 悪意あるWebページ | 訪問先DOMを制御 | 拡張権限を悪用 | RQ2 |

### 信頼境界
被スキャンコード = **常に未信頼**。エンジンは未信頼入力に対し有界時間・有界メモリで終了し、
判定を曲げられてはならない。拡張のホスト権限 `<all_urls>` は利用者の明示スキャン時のみ作用（主張）。

## 2. 評価の共通枠組み（methodology）

論文としての厳密性はここで担保する。全実験がこの共通規約に従う。

### 2.1 コーパスと正解（ground truth）
- **既存コーパス**: `samples/vulnerable`（13ファイル・6言語・**50 findings**、内訳は
  [EVALUATION.md](../docs/EVALUATION.md) E2/E3 に固定）、`test_problem`（1ファイル・20 findings）、
  `samples/safe`（**0 findings** 期待＝FP基準）。
- **正解ラベル**: 各サンプルの意図された脆弱性クラス（rule族）を `_results/ground-truth.json` に
  明示固定。検出は「同一ファイル × 行±L（L=2、EVALUATIONのSAST比較と同じ許容）× rule族一致」で正解と突合。
- **変異版の脆弱性保存（B系）**: 変換は**構成的に意味保存**（カタログ化された equivalence、
  §3 B1）であることを主担保とし、抜き取りで**最小実行PoC**（脆弱性が生きている実証）を付す。

### 2.2 ベースライン（横断比較）
| ツール | 種別 | 役割 | 既存資産 |
|---|---|---|---|
| **VibeGuard** | 正規表現 | 主対象 | `apps/cli` |
| **Bandit** | AST（Python） | 同一敵対条件での比較 | `scripts/sast-baseline-eval.mjs` 再利用 |
| **Semgrep** | パターン+データフロー（多言語） | パターン系の別実装比較 | 同上 |
| **CodeQL**（P2・任意） | データフロー解析 | 「データフロー型は回避に強いか」の対照 | 新規（重いので任意） |

ツール版・ルールパック版・実行日を必ず記録（Semgrep `p/default` は時間で変動するため）。

### 2.3 指標の形式定義
- 検出 detect(tool, vuln) = 上記突合規則を満たす finding が存在 ⇔ 1。
- **再現率** R(tool) = (Σ_v detect) / |GT|。
- **回避率（evasion rate）** 変換 t・ツール k に対し
  `ER(t,k) = |{v : detect(k,v)=1 ∧ detect(k, t(v))=0}| / |{v : detect(k,v)=1}|`
  （＝元は検出できたのに変換後に消えた割合＝相対再現率低下）。
- **可用性増幅（ReDoS, A1）**: 入力長 n に対するスキャン時間 T(n) を計測し、
  多項式/指数フィットで計算量クラス（線形/二次/指数）を判定。`amplification = T(n)/T(baseline)`。
- **FP率**（context機構の評価, B3/A2）: `samples/safe` 上の finding 数（0 が基準）。
- 統計処理: 時間系は**3回以上の中央値**（EVALUATION E5 と同じ）、分類系は決定的なので単発でよいが
  乱数を使う生成（B2）は seed 固定または試行数を明記。

### 2.4 参照環境・再現性
EVALUATION.md と同一（Win11 / i5-12400 / 32GB / Node v24）。各実験は**番号付きスクリプト
＋期待値**（`scripts/sec-*.mjs`）で third-party 再現可能にする。出力は `_results/` に JSON で固定。

## 3. 実験項目（RQ対応・8項目）

各項目: **RQ / 仮説（反証可能形）/ 手法 / 指標 / ベースライン / 成果物 / 優先度**。
優先度 P0（中心命題を支える必須）＞ P1（あると強い）＞ P2（余力）。

### Track A — サービス自身の安全性

#### A1. ReDoS / 計算量耐性 〔P0・RQ1〕
- **仮説 H1**: VibeGuard の47正規表現のうち ≥1 本が、長さ n の悪性入力に対し super-linear
  （二次以上）の T(n) を示し、`runRegex` に時間/サイズ上限が無いため単一ファイルで
  DESIGN §11.1 の性能契約（3s/1s/5min）を破れる。**反証条件**: 全ルールが線形かつ契約内。
- **手法**: (1) 全ルール正規表現を静的抽出し `recheck`/`vuln-regex-detector` で危険判定 →
  (2) 攻撃文字列生成 → (3) n を増やし CLI で T(n) 実測・計算量フィット →
  (4) **同手法を Semgrep の `p/default` ルール群にも適用**し「SAST一般の自己攻撃面」かを判定（C3）。
- **指標**: super-linear ルール数/47、契約を破る最小 n、amplification、（横断）ツール別 super-linear 率。
- **ベースライン**: Semgrep ルール集合（CodeQL は P2）。
- **成果物**: `a1-redos/` 攻撃コーパス＋ n–T 表＋計算量分類＋緩和案（per-match タイムアウト/長さキャップ/安全書換）。

#### A2. ゼロ送信（no-egress）主張の実証 〔P0・RQ2〕
- **仮説 H2**: PRIVACY.md の「ネットワークアクセス無し」は、静的（シンク不在）＋動的
  （キャプチャでリクエスト0）の**反証可能手続き**で4チャネル全て検証できる。**反証条件**: いずれかで egress>0。
- **手法**: (1) 静的＝CLI dist/Chrome dist/VS Code vsix にネットワークシンク無し（grep＋依存グラフ＋
  Chrome manifest の送信経路）。(2) 動的＝プロキシ監視/遮断下（mitmproxy/Fiddler/`--inspect`）で
  サンプル走査し外向き0観測。**方法論の一般化**: 同手続きを Bandit/Semgrep にも適用し
  「SASTのプライバシー主張を反証可能に検証する再利用可能プロトコル」として提示。
- **指標**: 静的シンク数（0基準）、動的外向きリクエスト数（チャネル×シナリオ表、0基準）。
- **成果物**: `a2-egress/` 静的監査＋キャプチャログ＋再現手順（検証プロトコルとして文書化）。

#### A3. Chrome 拡張の権限・攻撃面監査 〔P1・RQ2〕
- **仮説**: MV3 権限（`activeTab`/`scripting`/`<all_urls>` 等）は最小権限で、悪意ページからの
  権限/コード奪取経路が無い。**反証条件**: 不要権限 or 未検証の信頼境界クロス or `eval` 使用。
- **手法**: manifest 権限の必要性マッピング、background/side-panel/content のメッセージ経路レビュー、
  `storage.session` ハンドオフのライフタイム、CSP・`eval`/`Function` 不使用、`<all_urls>` の遅延読取確認。
- **指標**: 過剰権限数 / 未検証境界クロス数 / 危険API使用数（全0目標）。
- **成果物**: `a3-chrome-perms/` 権限表＋データフロー図＋指摘。

#### A4. サプライチェーン / 依存・Action 監査 〔P1・RQ2〕
- **仮説**: 配布物のサプライチェーンリスクは管理下（既知脆弱依存・過剰Action権限・未ピン依存が無い）。
- **手法**: `npm audit`/SBOM(CycloneDX)、`action.yml`・workflow の `permissions:` 最小化、
  3rd-party Action の SHA ピン、リリース成果物の provenance。
- **指標**: high+ 脆弱依存数 / 過剰権限workflow数 / 未ピンAction数。
- **成果物**: `a4-supplychain/` SBOM＋監査表。
- **備考**: 既存 self-scan（VibeGuard自身のルールで自走査）とは軸が異なる（依存・権限・出所）。

#### A5. 未信頼入力ファジング（堅牢性） 〔P1・RQ1〕
- **仮説**: 巨大/バイナリ/不正UTF-8/深いネスト等でもエンジンは無限ループ/未捕捉例外/メモリ枯渇を
  起こさず graceful に終了。A1（時間計算量）と相補（A5=堅牢性全般）。
- **手法**: CLI 入口に生成/変異ファジング（巨大行・NUL・無効UTF-8・深いネスト・100MB級）。
- **指標**: クラッシュ/ハング/未捕捉例外数・最大メモリ・回復可否。
- **成果物**: `a5-fuzz/` コーパス＋失敗ケース＋修正提案。

### Track B — 敵対的検出耐性

#### B1. 意味保存変換による回避（中心実験） 〔P0・RQ3〕
- **仮説 H3**: 意味保存変換の集合 𝒯 が存在し、平均 `ER` が有意に大きい（例 >0.3）。回避のしやすさは
  ツール種別に依存し、**正規表現系（VibeGuard）> データフロー系**の傾向を示す。**反証条件**:
  全変換で ER≈0、またはツール種別で差が無い。
- **手法**: (1) 変換 **taxonomy** を定義（字句: 文字列連結/エンコード/空白・コメント挿入、
  名前解決: エイリアス・`getattr`/動的属性、構造: 等価書換）。(2) `samples/vulnerable`＋`test_problem`
  の各脆弱性に変換を機械適用し変異コーパス生成（＝C1の artifact）。(3) **VibeGuard・Bandit・Semgrep**で
  元/変異の検出を突合し `ER(t,k)` を算出（C2）。(4) 変換×ツールのヒートマップ。
- **指標**: 変換別・ツール別 `ER`、全体再現率低下（before→after）、ツール種別間の ER 差。
- **ベースライン**: Bandit（py）, Semgrep（多言語）, CodeQL（P2）。
- **成果物**: `b1-evasion/` 変換カタログ＋変異コーパス＋ ER ヒートマップ。論文の主表/主図。

#### B2. AI生成の敵対コーパス 〔P1・RQ3〕
- **仮説**: 「検出器を通る脆弱コードを書け」と LLM に指示すると、素の脆弱コードより検出率が
  有意に下がる現実的コードが得られ、その回避手口は B1 taxonomy に概ね写像できる。
- **手法**: LLM に (a) 素の脆弱コード / (b) 回避指示つき を各 N 生成（seed/試行数記録）。脆弱性は
  人手＋簡易実行で確認。検出率差を検定し、回避手口を B1 分類に対応付け。生成は自リポ検証目的限定。
- **指標**: 素 vs 回避指示の検出率差（有意性）、手口の B1 taxonomy 被覆率。
- **成果物**: `b2-ai-adversarial/` 生成プロンプト＋コーパス＋検出結果。

#### B3. context-confidence / suppression の悪用 〔P0・RQ4〕
- **仮説 H4**: 本物の高リスク脆弱性を「テスト/フィクスチャ風パス」や「docstring偽装」に置くと、
  context-window 補正が confidence を action 閾値未満へ格下げし、**critical を含めて隠蔽**できる。
  かつ severity 依存の格下げ抑止を入れると、**FP削減効果（`samples/safe`=0維持）を損なわずに**
  隠蔽を防げる。**反証条件**: 偽装で格下げが起きない、または緩和案でFPが増える。
- **手法**: (1) 同一脆弱性を 通常パス vs 偽装パス（`*/tests/*`・`*_mock.*`・`conftest.py`）に置き
  confidence 比較（[`confidence.ts`](../packages/analyzer-core/src/confidence.ts) 逆用）。
  (2) docstring/ブロックコメント内の**実行されるコード**を紛れ込ませる言語別トリックで格下げ誘発。
  (3) **緩和案を実装し A/B**: 「critical/high は test パスでも格下げしない」適用前後で、
  隠蔽成功数（↓期待）と `samples/vulnerable` の confidence 分布・`samples/safe` FP（不変期待）を測る。
- **指標**: 格下げで閾値を割れた脆弱性数/試行、緩和前後の隠蔽成功数・FP数・confidence分布。
- **成果物**: `b3-suppression-abuse/` 攻撃ケース＋緩和実装＋ A/B 結果（C4）。

## 4. 妥当性への脅威（Threats to Validity）

- **構成概念（construct）**: 「検出」を行±2・rule族一致で定義 → 過大/過小評価しうる。意味一致でなく
  位置一致である旨を明記（EVALUATIONのSAST比較と同じ既知の制約）。「脆弱性保存」は構成的保証＋抜取りPoCで担保。
- **内部（internal）**: ツール版・ルールパック版・実行日で結果が動く（特に Semgrep `p/default`）→ 全て記録・固定。
  ReDoS時間はハード依存 → 中央値・参照環境明記、計算量**クラス**（線形/二次/指数）で報告し絶対秒数に依存しない。
- **外部（external）**: コーパスが小規模・著者作 → 一般化に限界。B1の変異母体に既存 OSS 由来
  （E6拡張の11リポ）も加え、母集団を広げる選択肢を P1 で用意。
- **結論（conclusion）**: B2 の生成は乱数依存 → seed/試行数固定と検定で過剰主張を防ぐ。横断比較は
  「coverage complementarity であり precision/recall の優劣ではない」という EVALUATION の留保を踏襲。

## 5. 成果物・再現性（artifact）

- 各 RQ → 番号付きスクリプト `scripts/sec-a1..`, `sec-b1..` ＋期待値（E1–E6 と同じ追跡可能性）。
- **公開アーティファクト**: B1 の変換 taxonomy ＋変異コーパス（C1）、検証プロトコル（A2, C2）、

- 参照環境・各ツール版・コーパスのコミットハッシュを `_results/manifest.json` に記録。

## 6. 優先度サマリ・推奨順

| 項目 | RQ | 優先度 | 工数 | 推奨理由 |
|---|---|---|---|---|
| B1 回避（横断） | RQ3 | **P0** | 中〜大 | 最も新規性が高い。C1+C2＝論文の主表。ベースライン必須 |
| B3 confidence悪用＋緩和A/B | RQ4 | **P0** | 中 | 自機構の二面性を測り緩和まで（C4）。差別化の核 |
| A1 ReDoS（横断） | RQ1 | **P0** | 中 | 「探す正規表現が攻撃面」という自己言及的知見（C3） |
| A2 ゼロ送信（プロトコル化） | RQ2 | **P0** | 小〜中 | 低工数で主張を反証可能に。検証プロトコルとして一般化 |
| B2 AI敵対コーパス | RQ3 | P1 | 中 | B1を「AI生成」文脈で補強（本サービスの動機と直結） |
| A5 ファジング | RQ1 | P1 | 中 | A1の相補・堅牢性網羅 |
| A3 Chrome権限 | RQ2 | P1 | 小 | MV3最小権限の確認 |
| A4 サプライチェーン | RQ2 | P1 | 小 | SBOM/audit で軽い |
| CodeQL ベースライン | RQ1/3 | P2 | 大 | 「データフロー型は回避/ReDoSに強いか」の対照群 |

**推奨**: **P0 4本（B1・B3・A1・A2）＝ RQ1–4 を1本ずつ被覆**。これで
「自己攻撃面（可用性 A1 / 機密性 A2）× 検出の限界と自機構の堅牢性（B1 / B3）」が、
**他SASTとの横断比較・再利用可能アーティファクト付き**で揃い、単独ツール監査でない一般化された
知見になる。P1/P2 は被験母集団・対照群の拡張として段階的に足す。

## 7. 倫理・scope境界

- 全実験は **VibeGuard 自リポジトリ（自サービス）＋公開SASTツール**を対象。第三者の本番システム・
  配信基盤・実利用者調査は含まない（**Out of scope**）。
- 攻撃PoCは「ハング再現」「検出漏れ提示」までに留め、武器化エクスプロイトは公開しない。
- LLM生成（B2）は自リポ検証目的に限定し、生成物を外部投入しない。
